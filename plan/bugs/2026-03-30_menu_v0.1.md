# Menu Component Bug Report (v0.1)

Based on Storybook testing and deep code review of `@thanh-libs/menu`.

**Total bugs: 13** — 4 Critical, 5 Major, 4 Minor

---

## 🔴 Critical (Feature hoàn toàn hỏng)

### 1. Popover Mode Functions as Inline
**File**: `MenuSubContent.tsx`

When setting `<Menu mode="popover">`, the `MenuSub` components do not render as floating popovers. Instead, they expand downwards inline, pushing the rest of the menu items down. Functionally and visually, there is currently no difference between `inline` and `popover` modes.

**Root cause**: `MenuSubContent` **always** renders `InlineSubContentStyled` regardless of `resolvedMode`. Even though `MenuSubPopover` sets up Floating UI (useFloating, useHover, useDismiss…), the `MenuSubContent` component **never** uses `FloatingPortal`, `setFloating` ref, or `floatingStyles`.

**Fix direction**: `MenuSubContent` must check `resolvedMode` from context. When `popover` → render in `FloatingPortal` + use `setFloating` + `floatingStyles` + `getFloatingProps`.

### 2. Hover Trigger Does Not Open Sub-menu
**File**: `MenuSubTrigger.tsx`

In the "Popover Sub-menus" story, hover triggers do not work. Hovering over a `MenuSubTrigger` does not open the floating sub-menu. The sub-menu only expands when clicking on it (and even then, it expands inline, as noted above).

**Root cause**: `MenuSubTrigger` only attaches a plain `onClick` handler. It **never** uses `getReferenceProps` or `setReference` from Floating UI context. Although `MenuSubPopover` sets up `useHover(floating.context, {...})`, the interaction props **are not passed down** to the trigger element.

**Fix direction**: When `resolvedMode === 'popover'`, trigger must use `setReference` ref + `getReferenceProps()` so that Floating UI interaction hooks (hover/click) can work.

---

### 3. `display="icon"` Does Not Hide Text Labels
**Files**: `MenuItem.tsx`, `MenuSubTrigger.tsx`, `styled.tsx`

Setting `<Menu display="icon">` fails to hide the text labels of the `MenuItem`s. The menu retains its full standard width and displays both the icon and the text label, entirely failing the purpose of the "Icon-Only Display (Mini Sidebar)".

**Root cause**: `MenuItem` and `MenuSubTrigger` **do not read** `display` from context. `MenuItemLabelStyled` has no prop to hide text. No styled component handles `display === 'icon'`.

**Fix direction**: Pass `display` down via context. `MenuItemLabelStyled` should accept an `ownerIconOnly` prop that hides label text. `MenuItemShortcutStyled` and `SubArrowStyled` should also hide in icon mode.

### 4. React Render-phase State Update Error
**File**: `MenuSub.tsx` — `MenuSubInline.registerSelected()`

**Error**: `Cannot update a component (MenuSubInline) while rendering a different component`

**Root cause**: In `registerSelected()`, `setOpen(true)` and `parentSub.registerSelected()` are called **inside** a `setSelectedCount` updater function. This triggers state updates within a state updater, which React 18+ strict mode detects and warns about.

```tsx
// MenuSub.tsx — registerSelected
setSelectedCount((c) => {
  const next = c + 1;
  if (next === 1 && !hasAutoExpandedRef.current) {
    setOpen(true);   // ❌ setState inside setState updater
  }
  if (next === 1 && parentSub) {
    parentUnregisterRef.current = parentSub.registerSelected(); // ❌ parent setState inside updater
  }
  return next;
});
```

**Fix direction**: Use `queueMicrotask` or `useEffect` to defer `setOpen(true)` instead of calling it directly in the updater.

## 🟡 Major (Hành vi sai / thiếu kết nối)

### 5. `floatingSettings` from Menu Context is Ignored
**File**: `MenuSub.tsx` (line 218–223)

`Menu` accepts a `floatingSettings` prop and passes it via context. But `MenuSubPopover` **hardcodes** `placement: 'right-start'` and **never reads** `floatingSettings`.

```tsx
const floating = useFloatingPosition({
  open: isOpen,
  onOpenChange: setOpen,
  placement: 'right-start',  // ❌ Hardcoded, ignores floatingSettings
});
```

**Fix direction**: Read `floatingSettings` from MenuContext and merge into `useFloatingPosition` options (placement, offset, flip, shift).

---

### 6. Ref Handling — Wrong Pattern, Breaks Callback Refs
**Files**: `Menu.tsx` (line 32), `MenuSubTrigger.tsx` (line 20)

```tsx
const containerRef = (externalRef as React.RefObject<HTMLDivElement>) || internalRef;
```

**Problem**: `externalRef` from `forwardRef` can be a **callback ref** (function), not a `RefObject`. Casting it as `RefObject` will **silently break** if the consumer uses a callback ref. Also, when `externalRef` is provided, the internal ref and external ref are **not synced**.

**Fix direction**: Use `useMergeRefs` from `@floating-ui/react` (already a dependency) or a custom hook to properly merge refs.

---

### 7. `tabIndex` Logic — Breaks Roving Tabindex Pattern
**File**: `MenuItem.tsx` (line 65)

```tsx
tabIndex={disabled ? -1 : undefined}
```

**Problem**: When `disabled` = false, `tabIndex` = `undefined`. But **all** non-disabled items should default to `tabIndex={-1}`, with only the focused item having `tabIndex={0}` (roving tabindex pattern). Currently, items without an explicit `tabIndex` **join the browser's default tab order**, meaning users can Tab through **every single item** instead of Tab-into-menu then ArrowDown.

**Fix direction**: Default `tabIndex={-1}` for all items, let `useMenuKeyboard` manage which item gets `tabIndex={0}`.

---

### 8. `getVisibleMenuItems` — Misses Popover Portal Items
**File**: `helpers/index.ts` (line 7–13)

```tsx
export const getVisibleMenuItems = (container: HTMLElement): HTMLElement[] => {
  const all = Array.from(container.querySelectorAll(ITEM_SELECTOR));
  return all.filter(
    (el) => el.closest('[role="menu"][data-collapsed="true"]') === null
  );
};
```

**Problems**:
1. In **popover mode** (once fixed), `MenuSubContent` renders inside `FloatingPortal` **outside** the menu root container. `getVisibleMenuItems` queries **only within** `containerRef.current`, so it will **miss all items** in popover portals.
2. During collapse **animation**, items inside `InlineSubContentStyled` are still in the DOM with `data-collapsed="true"` but partially visible — fast ArrowDown during animation can navigate to collapsing items.

**Fix direction**: For popover mode, keyboard navigation scope should be per-level. Consider separate keyboard handler for popover content.

---

### 9. CSS/Styled-Component — Missing Implementations
**Files**: `styled.tsx`, `MenuItemLabelStyled`, `InlineSubContentStyled`

Because labels aren't hidden and sub-menus lack a proper floating portal/reference setup, there are missing implementations in the styled components connecting `display="icon"` and `mode="popover"` to the DOM:
- `MenuItemLabelStyled` has no `ownerHidden`/`ownerIconOnly` prop
- No `PopoverSubContentStyled` exists for floating content styling (min-width, shadow, bg, border-radius, z-index)
- `SubArrowStyled` doesn't differentiate between inline (▾/▴) and popover (▸) modes

> [!WARNING]
> The recent implementation added the props and types for these features, but the actual CSS logic (hiding labels in icon mode) and Floating UI logic (rendering floating portals for `MenuSubContent` when in popover mode) is either missing, disconnected, or failing to trigger.

## 🟢 Minor (Edge case / polish)

### 10. Incorrect Caret Icon in Popover Mode
**File**: `MenuSubTrigger.tsx` (line 86)

For items that have a sub-menu, the caret currently always points downwards (`▾`). If a sub-menu is supposed to fly out to the right (as a floating popover), the caret should ideally point right (`▸`) to indicate the direction it will open.

```tsx
<SubArrowStyled aria-hidden="true">{isOpen ? '▴' : '▾'}</SubArrowStyled>
// Should be: popover mode → '▸', inline → isOpen ? '▴' : '▾'
```

---

### 11. `toggle()` Instead of `setOpen(false)` — Stale Closure Risk
**File**: `MenuSubContent.tsx` (line 17–24)

When pressing Escape or ArrowLeft, the handler calls `toggle()` (which does `setOpen(!isOpen)`). If `isOpen` is captured in a stale closure, the toggle could **re-open** instead of closing.

**Fix direction**: Use `setOpen(false)` directly since the intent is always to close.

---

### 12. Missing `aria-selected` / `aria-current` for Selected Item
**File**: `MenuItem.tsx`

Per WAI-ARIA Navigation Menu pattern, the selected item should have `aria-current="page"` (for navigation) or `aria-selected="true"`. Currently only visual indicators (✓ check + bold) are used — **no ARIA attribute** tells screen readers which item is active.

---

### 13. Visual Separation — Addressed ✅
Initially, it was hard to visually distinguish the inline mode from the popover mode in the story. **Resolved** by placing them inside styled `<aside>` blocks with colored backgrounds for clarity in the Storybook code.
